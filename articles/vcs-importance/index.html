اگر روی پروژه ای کار می‌کنید و با متن سر و کار دارید،‌ فرقی نمی‌کند که تنهایید یا با چند نفر کار می‌کنید، پروژه ای که روی آن کار می کنید خورد است یا کلان،‌ اگر از یک Version Control System استفاده نمی کنید نه تنها به این معنی است که روش کارتان اشتباه است بلکه از بسیاری چیزها محرومید.

مشکل کار با متن

در کار با متن بالاجبار باید چند دفعه متن‌تان را تغییر بدهید، اشتباهاتتان را اصلاح کنید، قسمت‌هایی را حذف و یا اضافه کنید تا به چیزی که از آن راضی هستید برسید. با این همه تغییر ناچار به جایی می‌رسید که فکر می‌کنید شاید تغییراتتان درست نبوده، یا چیزی را حذف کرده‌اید که حالا به آن نیاز دارید. شاید هم تنها کسی نیستید که روی این متن کار می‌کنید و پس از مدتی باید کارتان را با کار دیگران ادغام کنید.

در همهٔ این موارد احتمال اینکه کارتان را از دست بدهید بسیار زیاد است. هر راه حلی هم که برای این مشکل در نظر بگیرید به خوبی استفاده از یک Version Control System (VCS) نیست. ذخیره کردن متن در VCS مزایایی دارد که می‌توان منجمله به چند مورد آتی اشاره کرد. اول اینکه چون کارتان روی یک سرور آنلاین ذخیره شده است، حتی اگر کارتان را از دست بدهید همیشه می‌توانید به فایل‌های روی سرور دسترسی داشته باشید. دیگر اینکه همراه با هر فایلی که ذخیره می‌کنید تاریخچهٔ تغییرات آن فایل را هم ذخیره می‌کنید. پس هر وقت از چیزی خوشتان نیامد می‌توانید به نسخه‌های قبلی آن فایل رجوع کنید. ‌در آخر اینکه استفاده از VCS کار در گروه را برایتان آسان می‌کند.

Version Control System

Version Control System یا سیستم کنترل نسخه ها مثل ماشین زمانی برای فایل‌های شما است. در اینجا نسخه به یک کاپی از فایل های شما در یک زمان مشخص گفته می شود. مثلاً بعد از چند ساعت فکر می کنید قسمتی از کارتان تکمیل شده و باید تغییرهاتان را جایی ذخیره کنید. به این کاپی از فایل های شما یک نسخه گفته می‌شود. ذخیره کردن فایل ها در VCS به نام commit کردن یاد می شود. هر commit نمایانگر یک نسخه از فایل‌هایتان در یک زمان مشخص است. از ابزارهای مختلف VCS می‌توان به git، subversion، mercury و bazaar اشاره کرد. هر یک از این ابزارها خصوصیات منحصر به فرد خود را دارند. یکی از این خصوصیات چگونگی ذخیره فایل‌ها روی سرور است. از این نگاه می‌توان به دو نوع مختلف مرکزی و غیرمرکزی اشاره کرد. در یک VCS مرکزی یا centralized هر گاه بخواهید از فایل‌هایتان یک commit ایجاد کنید باید فایل‌هایتان به سرور آپلود شوند. از این گونه VCS می‌توان به subversion اشاره کرد. نقطهٔ مقابل اینگونه VCS decenralized است. به این معنی که فایل های شما هم روی فایل سیستم خودتان، هم روی سرور قرار دارد. یک commit اول روی سیستم شما صورت می‌گیرد و هر وقت خواستید می‌توانید این نسخه را به سرور آپلود کنید. اینگونه VCS برای سیستم‌هایی که همیشه دسترسی به اینترنت ندارند ایده‌آل است. نمونهٔ مشهور اینگونه VCS git است. در این نوشته به معرفی همین ابزار پرداخته می‌شود.

Git

قبل از شروع کار با git باید با چند مفهوم ساده و اساسی آشنا شویم. نیازی نیست همهٔ این مفاهیم را به خاطر بسپارید. همین که آنها را مرور کنید کافی است.

کلمه توضیحات
respository به دایرکتوری ای گفته می‌شود که VCS تغییرات فایل‌ها را در آن دنبال می‌کند.
commit یک نسخه از فایل‌ها است. همراه با هر commit، زمان آن، شخصی که commit را انجام داده، یک عنوان و در صورت نیاز توضیحات هم با آن ذخیره می‌شود.
diff فرق بین دو commit است. اینکه چه فایل‌هایی تغییر کرده‌اند و این تغییرها در کدام خط ها آورده شده است.
log تاریخچهٔ تغییرها. زمان، نام انجام دهندهٔ‌ commit، عنوان هر commit و قیمت hash. hash یک قیمت طولانی و منحصر به فرد (یا یگانه) است که git به هر commit می‌دهد تا آن را از commitهای دیگر متمایز کند.
branch branch یک کاپی مجازی از repository در یک مقطع زمانی است.
stash محل ذخیرهٔ موقت تغییرها است. مثلاً وقتی می‌خواهید کارتان را نیمه کاره رها کنید و به کار دیگری بپردازید اما نمی‌خواهید تغییرهایتان را از دست بدهید از stash استفاده می‌کنید.
patch فرق بین دو commit به صورت متنی است که git می‌تواند از آن استفاده کند. مثلاً وقتی نمی‌خواهید کسی به آخرین نسخه از repository شما دسترسی داشته باشد ولی می‌خواهید فقط یک commit را به اشتراک بگذارید از patch استفاده می‌کنید.

شروع کار با git

اولین باری که می خواهیم از git استفاده کنیم باید به git بگوییم که چی کسی هستیم. برای این کار باید نام و ایمیل خود را set کنیم.

(نکته: خطهایی که با کرکتر $ شروع می‌شوند دستوری را نشان می‌دهند که وارد شده است. خطهای دیگر خروجی دستور اند.)

$ git config --global set user.name 'John Carter'
$ git config --global set user.email 'carter.john@gmail.com'

اول یک دایرکتری برای پروژه درست می کنیم

$ mkdir -p workspace/secret/project
$ cd workspace/secret/project

یک git repository برای اولین بار به این صورت شروع می شود

$ git init .
Initialized empty Git repository in /home/barsoom/workspace/secret/project

حالا یک فایل ایجاد می کنیم و داخل آن چیزی می نویسیم

$ touch a.txt
$ echo 'I am John Carter from Barsoom' > a.txt
$ cat a.txt
I am John Carter from Barsoom

قبل از commit کردن فایل ها باید آنها را به repository اضافه کنیم (در واقع فایل‌ها به مکان staging اضافه می‌شوند)

$ git add .

برای هر commit باید یک عنوان (و یا پیام) بنویستم. نوشتن عنوان اجباری است و بدون آن، git به ما اخطار می دهد.

$ git commit -m 'announcing myself'
[master (root-commit) e7ee941] announcing myself
1 file changed, 1 insertion(+)
create mode 100644 a.txt

دیدن تاریخچه و فرق بین تغییرها

برای وضعیت فعلی تغییرات در دایرکتری فعلی از دستور status استفاده می‌کنیم

$ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)

        the_blue_light.txt

nothing added to commit but untracked files present (use "git add" to track)

برای دیدن تاریخچهٔ commit هایی که انجام داده ایم می توانیم از git log استفاده کنیم

$ git log
commit e7ee9410111f50348af8acf6ce1fca624445f828 (HEAD -> master)
Author: John Carter <carter.john@email.com>
Date:   Sun Sep 23 20:33:57 2018 +0430

    announcing myself

git log همراه با دیگر مشخصات هر commit یک قیمت hash نیز به ما می‌دهد. هر گاه نیاز به hash یک commit داشتیم لازم نیست همه این قیمت را کاپی کنیم. فقط به شش یا هفت کرکتر اول آن نیاز داریم. برای اینکه مجبور نباشیم هر دفعه که به hash یک commit نیاز داریم شش/هفت کرکتر را بشماریم و کاپی کنیم از --abbrev-commit استفاده می کنیم

$ git log --abbrev-commit
commit e7ee941 (HEAD -> master)
Author: John Carter <carter.john@email.com>
Date:   Sun Sep 23 20:33:57 2018 +0430

    announcing myself

حالا تغییرات بیشتری به فایل می آوریم

$ cat a.txt
I am John Carter from Barsoom.

Dear Ned,

I hope this letter finds you well. I am writing this letter to you from my death bed.

قبل از commit کردن،‌ می توانیم ببینیم چه تغییراتی به فایل ها آمده است. چند خط اول خروجی به ما کمک می کنند بفهمیم که چه تغییراتی مربوط به یک commit است. در خروجی زیر خطهایی که با + شروع می شوند تغییرهایی هستند که بعد از commit به وجود آمده اند.

$ git diff
diff --git a/a.txt b/a.txt
index bdd6876..ad21df1 100644
--- a/a.txt
+++ b/a.txt
@@ -1 +1,5 @@
 I am John Carter from Barsoom.
 +
 +Dear Ned,
 +
 +I hope this letter finds you well. I am writing this letter to you from my death bed.

حالا این تغییرها را هم commit می کنیم

$ git add .
$ git commit -m 'letter intro'
[master 5025a43] letter intro
 1 file changed, 4 insertions(+)

حالا در این repo دو commit داریم. اگر بخواهیم فرق بین هر دو را ببینیم اول باید hash هر commit را بدانیم

$ git log --abbrev-commit
commit 5025a43 (HEAD -> master)
Author: John Carter <carter.john@email.com>
Date:   Sun Sep 23 20:47:12 2018 +0430

    letter intro

commit e7ee941
Author: John Carter <carter.john@email.com>
Date:   Sun Sep 23 20:33:57 2018 +0430

    announcing myself

برای دیدن تغییرات این دو commit

$ git diff 5025a43 e7ee941
diff --git a/a.txt b/a.txt
index ad21df1..bdd6876 100644
--- a/a.txt
+++ b/a.txt
@@ -1,5 +1 @@
 I am John Carter from Barsoom.
 -
 -Dear Ned,
 -
 -I hope this letter finds you well. I am writing this letter to you from my death bed.

کار با branch

فرض کنید به فکر ایده‌ای می‌افتید و می‌خواهید آن را امتحان کنید اما در عین حال باید کاری را که محول شده‌اید انجام دهید. به جای اینکه از فایل‌هایتان کاپی بگیرید می‌توانید یک branch ایجاد کنید، تغییراتی که می‌خواهید روی آن branch اعمال کنید و هر گاه که خواستید به branch اصلی برگردید و کارتان را پیش ببرید. اگر دیدید ایده‌ای که به آن فکر کردید موفق بوده می‌توانید دو branch را با هم merge کنید. اگر هم دیدید تغییراتی که آورده‌اید موفق نبوده می‌توانید به راحتی branch را حذف کنید. در git branch اصلی به نام master یاد می‌شود. برای اینکه branch تازه‌ای ایجاد کنید می‌توانید از دستور زیر استفاده کنید

$ git checkout -b jarsoom
Switched to a new branch 'jarsoom'

بعد از اینکه تغییرهایتان را به این branch آوردید می‌توانید به راحتی به master branch برگردید

$ git checkout master
On branch master
nothing to commit, working tree clean

و برای merge کردن دو branch

$ git merge jarsoom
Updating 6a7245a..42522a8
Fast-forward
 tharks.txt | 1 +
 1 file changed, 1 insertion(+)
 create mode 100644 tharks.txt

هر وقت خواستید یک branch را حذف کنید از این دستور استفاده کنید

$ git branch -d jarsoom
Deleted branch jarsoom (was 42522a8).

استفاده از stash

فرض کنید در branch فعلی مقداری کار کرده‌اید و به یاد این می‌افتید که باید کار دیگری را شروع کنید. در عین حال نمی‌خواهید این مقدار کاری را که انجام داده‌اید از دست بدهید. می‌توانید از stash برای ذخیره این تغییرات استفاده کنید.

$ echo "a new source of energy" > the_blue_light.txt
$ add . 
$ git stash push -m 'about the blue light' .
Saved working directory and index state On master: about the blue light

stash تغییرات شما را به صورت یک stack ذخیره می‌کند یعنی هر وقت تغییراتتان را به stash ‍push می‌کنید، این تغییرات در نقطهٔ بالایی (و یا شروع)‌ این stack قرار می‌گیرند. برای دیدن لیست stash از این دستور استفاده کنید

$ git stash list
stash@{0}: On master: about the blue light

و برای اعمال یک stash از این دستور استفاده کنید

$ git stash apply stash@{0}
On branch master
Changes to be committed:
  (use "git reset HEAD <file>..." to unstage)

        new file:   the_blue_light.txt

ایجاد و اعمال patch

خروجی git diff patch نام دارد.

$ git diff --staged
diff --git a/the_blue_light.txt b/the_blue_light.txt
new file mode 100644
index 0000000..df0a0ef
--- /dev/null
+++ b/the_blue_light.txt
@@ -0,0 +1 @@
+a new source of energy

git diff --staged > mars_patch.txt

در کد بالا از پارامتر --staged به این خاطر استفاده کرده‌ایم که فایل‌هایمان را به staging اضافه کرده‌ایم. اگر هنوز فایل‌هایتان را به staging اضافه نکرده‌اید می‌توانید این پارامتر را در نظر نگیرید. اگر خروجی بالا را دریک فایل ذخیره کنید، به این فایل patch گفته می‌شود. می‌توانید این فایل را به کسی بفرستید که یک کاپی از repository شما دارد اما به آخرین نسخهٔ فایل‌ها دسترسی ندارد. تنها کاری که باید برای اعمال این patch انجام بدهید اجرای این دستور است

$ git apply mars_patch.txt
$ git status
On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)

        the_blue_light.txt

nothing added to commit but untracked files present (use "git add" to track)

آپلود به server

در git به سرور remote گفته می‌شود. هر repository می‌تواند چند remote داشته باشد. اگر فرض کنیم در جایی مثل github یا bitbucket یک repository ایجاد کرده‌اید با آدرس git@github.com:johncarter/mars-diary.git، می‌توانید این remote را به این صورت به repository خود اضافه کنید

$ git remote add origin git@github.com:johncarter/mars-diary.git

در اینجا remote ای که اضافه کرده‌ایم origin نام دارد.

کسی که می‌خواهد برای اولین بار با این repository کار کند باید آن را clone کند

$ git clone git@github.com:johncarter/mars-diary.git mars-diary
Cloning into 'mars-diary...
done.

برای دیدن لیست remote های یک repository

$ git remote -v
origin  git@github.com:johncarter/mars-diary.git (fetch)
origin  git@github.com:johncarter/mars-diary.git (push)

آپلود کردن فایل‌ها به git با نام push یاد می‌شود. با هر push باید نام remote (که به نام upstream هم یاد می‌شود) را بعد از پارامتر -u و branch ای که می‌خواهیم فایل‌ها به آن آپلود شوند را ذکر کنیم.

$ git push -u origin master
Enumerating objects: 9, done.
Counting objects: 100% (9/9), done.
Delta compression using up to 4 threads
Compressing objects: 100% (6/6), done.
Writing objects: 100% (9/9), 850 bytes | 850.00 KiB/s, done.
Total 9 (delta 0), reused 0 (delta 0)
To git@github.com:johncarter/mars-diary.git
 * [new branch]      master -> master
Branch 'master' set up to track remote branch 'master' from 'dir'.

اگر بخواهیم تغییرات را از upstream دانلود کنیم باید از دستور pull استفاده کنیم. درست مثل دستور push در pull هم باید نام remote و branch را ذکر کنیم و تفاوت این دستور با push این است که نیازی به ذکر پارامتر -u نیست

$ git pull origin master
remote: Enumerating objects: 9, done.
remote: Counting objects: 100% (9/9), done.
remote: Compressing objects: 100% (6/6), done.
remote: Total 9 (delta 0), reused 0 (delta 0)
Unpacking objects: 100% (9/9), done.
From git@github.com:johncarter/mars-diary.git
 * branch            master     -> FETCH_HEAD
 * [new branch]      master     -> origin/master

در این نوشته فقط با چند دستور ابتدایی کار با git آشنا شدیم. در نوشته‌های بعد با ترفندهای کارآمدتر و پیشرفته تری آشنا خواهیم شد. مثلاً اینکه چطور از یک branch فقط یک فایل را بیرون بکشیم و یا چطور از یک commit مشخص یک branch ایجاد کنیم.
